# Technical Design Notes

Этот файл служит локальной архитектурной документацией проекта.

Он не заменяет системные инструкции среды, но задает технический вектор для разработки `Darkfront: Last Stand` как наследника `Super Auto Pets`.

## 1. Project Summary

- Рабочее название: `Darkfront: Last Stand`
- Жанр: `auto battler`
- Главный референс: `Super Auto Pets`
- Формат MVP: короткий `arena-like` run с асинхронными боями
- Главная задача: воспроизвести качество core loop `SAP`, а не просто скопировать список механик

## 2. Product Goals

Игра должна:

- быстро объясняться
- оставаться глубокой за счет синергий
- поддерживать сессии длиной `10-20` минут
- быть хорошо читаемой в поражении и победе
- расширяться через контент, а не через бесконтрольное усложнение правил

## 3. MVP Scope

В MVP входит:

- один run до `10` побед или `5` поражений
- команда из `5` слотов
- магазин с `buy`, `sell`, `merge`, `freeze`, `roll`
- один боевой ряд
- детерминированный battle resolver
- ограниченный набор триггеров и эффектов
- `3-4` тира контента
- один основной thematic pack

В MVP не входит:

- real-time PvP
- тяжелая мета-прогрессия
- mana-система уровня `Unicorn Pack`
- toys и hard-mode toys
- множество режимов
- grid-based бой

## 4. Architectural North Star

Кодовая база должна позволять:

- полностью отдельно тестировать бой без UI
- воспроизводить одинаковый результат по одинаковому seed
- сериализовать состояние run
- расширять контент декларативно
- разбирать спорные бои через event log

Главный принцип:

`simulation first, presentation second`

## 5. Layered Architecture

### 5.1. Content Layer

Хранит статические данные:

- описания юнитов
- описания еды
- shop tier tables
- наборы триггеров
- балансные числа

Формат желательно держать как:

- `JSON`
- либо `TypeScript`-объекты с валидацией через `Zod`

### 5.2. Domain Layer

Сердце проекта. Здесь живут:

- сущности команды, юнита, магазина, run
- правила merge и опыта
- shop engine
- battle engine
- event queue
- effect resolver
- RNG abstraction

Этот слой не должен знать ничего о:

- `React`
- `DOM`
- `Canvas`
- `localStorage`
- сетевых запросах

### 5.3. Application Layer

Оркестрирует use cases:

- `startRun`
- `startTurn`
- `rollShop`
- `buyPet`
- `sellPet`
- `mergePets`
- `freezeSlot`
- `feedPet`
- `startBattle`
- `resolveRound`
- `saveRun`
- `loadRun`

### 5.4. Infrastructure Layer

Отвечает за:

- storage
- загрузку данных
- seed generation
- telemetry
- источник команд для асинхронного матчинга

### 5.5. Presentation Layer

Содержит:

- экраны
- battle playback
- drag and drop или click-to-swap
- hover/tooltips
- анимацию событий

UI только показывает и отправляет команды. Он не является носителем игровых правил.

## 6. Determinism Rules

Для авто-баттлера это критично.

Обязательные правила:

- весь случайный выбор идет через единый RNG-интерфейс
- никаких `Math.random()` внутри доменной логики
- battle log должен хранить ключевые roll results
- один и тот же input snapshot обязан давать один и тот же output

Это важно и для тестов, и для дебага, и для последующего replay.

## 7. Event Model

Вместо ad-hoc логики по месту лучше строить бой через явные события.

Минимальный набор событий для MVP:

- `TURN_STARTED`
- `SHOP_ROLLED`
- `PET_BOUGHT`
- `PET_SOLD`
- `PETS_MERGED`
- `FOOD_USED`
- `BATTLE_STARTED`
- `ABILITY_TRIGGERED`
- `DAMAGE_DEALT`
- `PET_HURT`
- `PET_FAINTED`
- `PET_SUMMONED`
- `BATTLE_ENDED`
- `ROUND_RESOLVED`

Хорошая практика:

- домен меняет state
- одновременно пишет нормализованный event log

## 8. Resolution Order

Самая опасная часть жанра это не контент, а неявный порядок резолва.

Для MVP он должен быть зафиксирован и покрыт тестами.

Пример каркаса:

1. `Start of battle`
2. выбор атакующих
3. одновременный обмен уроном
4. death check
5. faint triggers
6. summon resolution
7. empty-front checks
8. переход к следующему обмену

Если в проекте появляется новый trigger type, он обязан быть встроен в эту последовательность явно.

## 9. Content Authoring Rules

Каждый юнит должен описываться как данные, а не как отдельный кусок bespoke-кода.

Желательная структура:

- `id`
- `name`
- `tier`
- `baseStats`
- `tags`
- `triggers`
- `targeting`
- `effectsByLevel`

Антипаттерн:

- юнит требует отдельный `if (unit.name === "...")`

Если таких проверок становится много, значит архитектура контента недостаточно декларативная.

## 10. Recommended MVP Trigger Set

Чтобы не утонуть в сложности, в ранней версии проекта лучше поддержать только:

- `Buy`
- `Sell`
- `Start of turn`
- `End turn`
- `Start of battle`
- `Hurt`
- `Faint`
- `Friend summoned`
- `Level-up`

Этого уже достаточно для сильной игры.

Можно позже добавить:

- `Before attack`
- `After attack`
- `Knock out`
- `Enemy summoned`
- `Transform`

## 11. Balance Direction

В `SAP-like` проекте баланс не должен начинаться с бесконечного числа юнитов.

Лучше сначала добиться:

- нескольких рабочих архетипов
- стабильной ранней игры
- читаемого midgame power curve
- отсутствия одной доминирующей экономической линии

Приоритеты баланса:

1. Ясность решений
2. Темп матча
3. Разнообразие билдов
4. Красота отдельных карт

## 12. Testing Strategy

Минимальный обязательный набор тестов:

- merge и опыт
- level-up rewards
- freeze / roll / shop refresh
- порядок start-of-battle триггеров
- faint и summon chains
- ничьи и одновременные смерти
- детерминизм по seed

Полезные форматы тестов:

- unit tests для чистых правил
- snapshot tests для state transitions
- golden tests для целых боев

## 13. UX Requirements

Даже у сильного battle core игра развалится без хорошего UX магазина.

Обязательные UX-свойства:

- все основные действия в `1-2` клика
- мгновенный tooltip со способностью
- читаемые числа атаки и здоровья
- понятное отображение временных и постоянных баффов
- предсказуемое drag/drop или swap-поведение

## 14. Performance Requirements

MVP не обязан быть технически сложным, но обязан быть отзывчивым.

- shop actions должны ощущаться моментальными
- бой должен быстро симулироваться даже при длинных summon chains
- UI не должен пересчитывать доменную логику на каждом рендере

## 15. Non-Goals

Мы не оптимизируем проект на старте под:

- максимально широкий feature set
- ультра-сложный late game
- сетевую архитектуру реального времени
- все паковые механики SAP сразу

Правильный путь:

- сначала чистый playable core
- потом контент
- потом расширяющие системы

## 16. Working Rule For Future Docs

Документы в репозитории должны читаться как система:

- `GDD-MVP.md` объясняет игру и механику
- `SAP-UNITS.md` объясняет контентную библиотеку и дизайн-грамматику
- `CLAUDE.md` объясняет, как это реализовывать технически

Если новая информация не вписывается ни в один из этих уровней, лучше создать отдельный документ, чем раздувать текущие файлы до каши.
